The problem Nuro's FTL compiler was built to solve

Third-party compilers cause silent regressions. Where does yours start?

Nuro's engineers isolate FP32 subgraphs with segment breakers because TensorRT and FP16 paths drift without crashing. This tool runs your ONNX graph, compares every activation boundary between reference and candidate compile paths, and tells you exactly which node to break first.

"Third party compilers are notorious for causing silent regressions in the model's performance." — Nuro FTL blog

What this solves (that a segment diagram doesn't)

❌ Portfolio toy

Fake latency tables, self-consistency parity, YAML rules with no failing tensors.

✓ This tool

Augments your ONNX graph, dumps every intermediate activation, diffs reference vs FP16/optimized path, ranks divergences, outputs FTL-style breaker YAML for the first failure.

Workflow

  1. segment-receipts scan model.onnx — find first divergence
  2. Apply recommended break_before_nodes / force_fp32_nodes
  3. Re-scan until clean · optional plan/run for stitch receipt
